# 服务端测试目录 (tests/server)

> **路径**: `50-examples/50.6-ai-agent/ai-agent-production/tests/server/`

## 概述

此目录存放 **AI Agent 生产应用** 的服务端集成测试用例。与单元测试不同，这里的测试针对完整的 HTTP 服务器实例运行，验证路由、中间件、认证、业务逻辑在真实请求链路中的协同工作。测试框架采用 **Vitest**，HTTP 服务器基于 **Hono** 构建。

## 文件说明

| 文件 | 说明 |
|------|------|
| `api.test.ts` | API 服务端集成测试套件，覆盖健康检查、Agent 列表、工作流列表、认证拦截与 404 处理 |

## 测试覆盖范围

### 1. 健康检查 (`GET /health`)

验证服务器基础可用性：

- 响应状态码必须为 `200`
- 返回 JSON 中 `status` 字段为 `"ok"`
- `version` 字段符合预期版本号（如 `"1.0.0"`）

这是部署流水线中最常用的探针接口，也是 CI/CD 健康检查的基础。

### 2. Agent 管理接口 (`GET /api/agents`)

验证多 Agent 系统的注册与暴露：

- 返回的 `agents` 字段必须是数组
- 至少包含 4 个预定义 Agent：`researcher`（研究员）、`writer`（写手）、`reviewer`（审校）、`orchestrator`（编排器）

这些 Agent 构成一个完整的协作工作流：编排器接收任务后分派给研究员收集资料，写手生成内容，审校进行质量检查。

### 3. 工作流管理接口 (`GET /api/workflows`)

验证预置工作流的可用性：

- 返回的 `workflows` 字段必须是数组
- 至少包含 2 个预定义工作流：`techDoc`（技术文档撰写）、`codeReview`（代码审查）

工作流是 Agent 协作的编排模板，定义了任务在多个 Agent 之间的流转顺序与条件分支。

### 4. 认证拦截测试

验证未授权访问被正确拒绝：

| 接口 | 方法 | 预期状态码 | 说明 |
|------|------|------------|------|
| `/api/agents/invoke` | POST | 401 | Agent 调用接口需要登录态 |
| `/api/mcp/tools` | GET | 401 | MCP 工具列表接口需要登录态 |

这些测试确保敏感操作不会因为路由配置疏漏而暴露给匿名用户。

### 5. 404 处理

验证不存在的路由返回规范的 JSON 错误：

- 状态码 `404`
- 响应体包含 `error` 字段，且错误信息包含"不存在"等友好提示

良好的 404 处理有助于前端统一错误展示与日志分析。

## 测试架构说明

```ts
import app from "../../src/server/index.js";
```

测试直接导入 Hono 应用实例（而非启动真实 HTTP 端口），利用 Hono 的 `app.request()` 方法在内存中模拟 HTTP 请求。这种"无端口测试"的优势：

1. **速度快**：无需等待端口绑定与网络 IO，测试在毫秒级完成
2. **无冲突**：并行运行测试时不会因为端口占用而失败
3. **易集成**：可在任何环境中运行，包括 CI 容器与无网络沙箱

## 开发规范

1. **测试命名**：使用中文描述测试意图，`it("GET /health 应返回健康状态")`，便于在测试报告中快速定位失败用例。
2. **边界覆盖**：除 Happy Path 外，还应测试空数组、超长输入、特殊字符等边界条件。
3. **状态隔离**：每个测试用例应独立运行，不依赖其他测试的执行顺序或副作用。如有需要，使用 `beforeEach` / `afterEach` 重置数据库或缓存状态。
4. **Mock 外部依赖**：若测试涉及调用 LLM API（如 OpenAI、Claude），必须使用 `msw` 或 `vitest.fn()` Mock 网络层，避免：
   - 产生真实的 API 费用
   - 因网络波动导致测试不稳定
   - 泄露 API Key 到 CI 日志

## 扩展方向

- **`auth.test.ts`**：认证流程专项测试（登录、Token 刷新、权限校验、登出）
- **`agent-invoke.test.ts`**：Agent 调用接口的完整链路测试（含 Mock LLM 响应）
- **`mcp.test.ts`**：MCP（Model Context Protocol）工具注册与调用测试
- **`workflow.test.ts`**：工作流引擎测试，验证多 Agent 协作的顺序与异常回滚
